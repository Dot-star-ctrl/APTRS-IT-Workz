Vulnerability Details

Methodology

In this part of the report, the procedure, consisting of several phases, will be described. In addition, a risk model will also be formed based on the Common Vulnerability Scoring System (CVSS) to determine the level of the vulnerability.

Procedure

A procedure has been chosen to use for the execution of the penetration test. For this penetration test it was chosen to use Penetration Execution Standard (PTES) as the framework. By using this procedure, the pen test work can be divided into 7 phases and the pen test can be carried out carefully

Phase 1: Intelligence Gathering

This phase is also called 'reconnaissance'. In this phase, the aim is to collect as much useful information as possible from the target. OSINT tools such as nslookup, WHOIS and Maltego can be used for this. These tools help collect public information from the target. The information provided by the client, such as network drawings and IP number plans, can also be used in a later phase. With the help of this information, it should be possible to discover certain access possibilities to the company that can be used in subsequent phases. Some companies go wrong by unknowingly or knowingly making (sensitive) information public, which makes it easy for attackers to gain entry. The information that can be gathered from this phase.

Phase 2: Threat Modeling

In this phase, the information obtained from phase 1 is analysed. A SWOT analysis and/or risk analysis could possibly be compiled for this. By forming these analyses, it becomes (more) clear which attack methods could be effective and what the threat to the company could be.

Phase 3: Vulnerability Analysis

During this phase, potential vulnerabilities of systems and applications are analyzed. This involves testing using tools such as Nmap, Netdiscover or Nikto to see what possible vulnerabilities are.

Phase 5: Post Exploitation

In this phase, the value of the compromised machines is determined and whether it is possible to maintain control over the machine(s) by means of a possible backdoor. In this phase, it is also examined whether it would have been possible to obtain more rights to the systems within the network.

Phase 6: Reporting

In this phase, all findings are brought together in a report. This report describes the findings, explains the CVSS scoring system, addresses the possible impact of the vulnerabilities and provides advice for the company to resolve the vulnerabilities found.

Phase 7: Re-audit

Re-audit actually means re-testing. The vulnerabilities found are resolved on the basis of the recommendations, after which it is tested whether the vulnerabilities have actually been resolved. This ensures that the company has resolved the problems.

CVSS

The Common Vulnerability Scoring System provides a way to capture the key features of a vulnerability and create a numerical score that reflects its severity. The numeric score can then be converted into a qualitative representation such as low, medium, high and critical. This helps organizations properly assess and prioritize their vulnerability management processes.

Base Metric

When determining the score for Basic Data, it should be assumed that the attacker has advanced knowledge of the target system's vulnerabilities, including general configuration and standard defense mechanisms (e.g., built-in firewalls, rate limits, traffic monitoring). For example, exploiting a vulnerability with repeatable, deterministic success should still be considered a low value for attack complexity, regardless of the attacker's knowledge or capabilities. Furthermore, the mitigation of attacks specific to the target (e.g. custom firewall filters, access lists) should be reflected in the environmental score group.

Attack Vector(AV)

This metric represents the context in which exploitation of the vulnerability is possible. The value of this metric (and consequently the base score) will be greater the farther an attacker can get to exploit the vulnerable component. The list of possible values is shown in the table.

Metric value Description
Network (N) A vulnerability that can be exploited with network access means that the vulnerable component is bound to the network stack and the attacker's path passes through OSI layer 3 (the network layer).
Adjacent(A) A vulnerability that can be exploited with adjacent network access means that the vulnerable component is bound to the network stack, but the attack is limited to the same shared physical (e.g. Bluetooth, IEEE 802.11) or logical (e.g. local IP subnet) network , and cannot run across an OSI layer 3 boundary (e.g. a router)
Local (L) A vulnerability that can be exploited with local access means that the vulnerable component is not bound to the network stack, and the attacker can find its way through read/write/execute capabilities.
Physical (P) A vulnerability that can be exploited with physical access requires the attacker to physically touch or manipulate the vulnerable component.

Attack Complexity (AC)

This metric describes the conditions beyond the attacker's control that must exist for the vulnerability to be exploited. As described below, such circumstances may require gathering more information about the target, the presence of certain system configuration settings, or computer exceptions. The list of possible values is shown in the table.

Metric value Description
Low (L) There are no special entry conditions or extenuating circumstances. An attacker can expect repeatable success against the vulnerable component.
High (H) A successful attack depends on circumstances beyond the attacker's control. That is, a successful attack cannot simply be performed, but the attacker must make a measurable effort in preparation or execution against the vulnerable component before a successful attack can be expected. The attacker must perform target-specific reconnaissance. attacker must prepare the target environment to improve the reliability of the exploit.-The attacker must inject himself into the logical network path between the target and the resource requested by the victim to read and/or modify the network communication (e.g. man in the middle attack).

Privileges Required (PR)

This metric describes the level of privileges an attacker must have to exploit the vulnerability. The list of possible values is shown in table

Metric value Description
None (N) The attacker is unauthorized prior to the attack and therefore does not need access to settings or files to carry out an attack.
Low (L) The attacker is authorized with rights that provide basic capabilities that would normally only affect a user's settings and files.
High (H) The attacker has permissions that provide significant (e.g., administrative) control over the vulnerable component and can affect its settings and files.

User Interaction (UI)

This metric captures the requirement for a human user, other than the attacker, to participate in the successful compromise of the vulnerable component. The list of possible values is shown in the table.

Metric value Description
None (N) The vulnerable system can be exploited without user intervention.
Required (R) Successful exploitation of this vulnerability requires user action before the vulnerability can be exploited. For example, a successful exploit may only be possible during the installation of an application by a system administrator.

Scope (S)

The scope metric captures whether a vulnerability in a vulnerable component affects resources in components that are outside the security scope. The base score is greatest with a scope change. The list of possible values is shown in the table.

Metric value Description
Unchanged (U) An exploited vulnerability can only affect resources controlled by the same security authority. In this case, the vulnerable component and the affected component are either the same or both are managed by the same security authority.
changed (C) An exploited vulnerability can impact resources outside the security scope controlled by the security authority of the vulnerable component. In this case, the vulnerable component and the affected component are different and are managed by different security authorities.

Impact Metrics

The Impact metrics capture the effects of a successfully exploited vulnerability on the worst outcome component most directly and predictably associated with the attack.

Confidentiality (C)

This metric measures the impact on the confidentiality of the information resources controlled by a software component as a result of a successfully exploited vulnerability. Confidentiality refers to limiting access to and disclosure of information to only authorized users, as well as preventing access by or disclosure to unauthorized persons. The base score is greatest when the loss for the affected component is highest. The list of possible values is shown in table.

Metric value Description
High (H) There is a complete loss of confidentiality, revealing all resources within the affected component to the attacker. Another possibility is that only part of the limited information is accessed, but that the information released has a direct, serious impact. For example, an attacker steals the administrator's password or the private encryption keys of a web server.
Low (L) There is some loss of confidentiality. Some restricted information is accessed, but the attacker has no control over what information is obtained, or the extent or nature of the loss is limited. The disclosure of information does not result in an immediate, serious loss to the affected component
None (N) There is no loss of confidentiality within the affected component.

Integrity (I)

This metric measures the integrity impact of a successfully exploited vulnerability. Integrity refers to the reliability and truthfulness of information. The base score is highest when the consequence on the component in question is highest. The list of possible values is shown in table.

Metric value Description
High (H) There is a total loss of integrity, or a complete loss of protection. For example, the attacker could modify all files protected by the affected component. Another possibility is that only some files can be changed, but a malicious change would have a direct and serious effect on the affected component.
Low (L) Change of data is possible, but the attacker has no control over the consequences of a change, or the amount of change is limited. The change of the data has no direct, serious consequences for the affected component.
None (N) There is no loss of integrity within the affected part.

Availability (A)

This metric measures the impact on the availability of the affected component as a result of a successfully exploited vulnerability. This metric refers to the loss of availability of the affected component itself, such as a network service (e.g. web, database, email). The base score is greatest when the impact on the affected component is highest. The list of possible values is shown in the table

Metric value Description
High (H) There is a total loss of availability, which can completely deny the attacker access to resources in the affected component.
Low (L) Performance is degraded or there are interruptions in resource availability. Even if repeated exploitation of the vulnerability is possible, the attacker would not have the option to completely deny service to legitimate users.
None (N) There is no loss of integrity within the affected part.

A Temporal Metrics

The temporal data measures the current state of exploitation techniques or the availability of code, the existence of any patches or work arounds, or the confidence one has in the description of a vulnerability.

Exploit Code Maturity (E)

This metric measures the likelihood of the vulnerability being attacked and is usually based on the current state of exploitation techniques, the availability of exploit code, or active, inthe-wild exploitation. The list of possible values is shown in table. The more easily a vulnerability can be exploited, the higher the vulnerability score.

Metric value Description
Not Defined (X) Assigning this value to the metric does not affect the score. It is a score comparison signal to skip this statistic.
High (H) Functional autonomous code exists, or no exploit is needed (manual trigger) and details are generally available. Exploit code works in any situation, or is actively distributed through an autonomous agent (such as a worm or virus). Networked systems are likely to see scans or exploit attempts. Exploit program development has reached the level of reliable, widely available, easy-to-use automated tools.
Functional (F) Functional exploit code is available. The code works in most situations where the vulnerability exists.
Proof of Concept (P) Proof-of-concept exploit code is available, or an attack demonstration is impractical for most systems. The code or technique is not functional in all situations and may require significant modification by an experienced attacker.
Unproven (U) There is no exploit code available, or an exploit is theoretical.

Remediation Level (RL)

The remediation level of a vulnerability is an important factor in prioritization. A vulnerability is usually unpatched when it is first published. Workarounds or hotfixes may provide a workaround until an official patch or upgrade is released. The list of possible values is shown in table. The less official and permanent a fix, the higher the vulnerability score.

Metric value Description
Not Defined (X) Assigning this value to the metric does not affect the score. It is a score comparison signal to skip this statistic.
Unavailable (U) Either no solution is available or it is impossible to apply.
Workaround (W) An unofficial, non-vendor solution is available. In some cases, users of the affected technology will create their own patch or provide steps to circumvent or otherwise mitigate the vulnerability.
Temporary Fix (T) An official but workaround is available. This includes cases where the vendor releases an interim hotfix, tool, or workaround.
Official Fix (O) A complete supplier solution is available. The vendor has released an official patch or an upgrade is available.

Report Confidence (RC)

This metric measures the degree of confidence in the existence of the vulnerability and the credibility of the known technical details. Sometimes only the existence of vulnerabilities is disclosed, but without specific details. For example, an impact may be recognized as undesirable, but the underlying cause is not known. The list of possible values is shown in table. The more a vulnerability is validated by the vendor or other reputable sources, the higher the score

Metric value Description
Not Defined (X) Assigning this value to the metric does not affect the score. It is a score comparison signal to skip this statistic.
Confirmed (C) Detailed reports exist, or functional reproduction is possible (functional exploits may provide this). Source code is available to independently verify the research's claims, or the author or vendor of the affected code has confirmed the presence of the vulnerability.
Reasonable (R) Significant details are published, but researchers either don't have full confidence in the root cause, or don't have access to the source code to fully confirm all the interactions that could lead to the result. However, there is reasonable confidence that the bug is reproducible and that at least one effect can be verified (proof-of-concept exploits can provide this).
Unknown (U) There are reports of impacts that indicate the presence of a vulnerability. The reports indicate that the cause of the vulnerability is unknown, or the reports differ in opinion about the cause or consequences of the vulnerability. The reporters are unsure about the true nature of the vulnerability and there is little confidence in the validity of the reports and whether a static Base Score can be applied given the differences described.

Environmental Metrics

These measurements allow the analyst to tailor the CVSS score to the importance of the affected IT asset to a user's organization, measured in terms of the presence of complementary/alternative security controls, Confidentiality, Integrity, and Availability.

Security Requirements (CR, IR, AR)

These metrics allow the analyst to tailor the CVSS score to the importance of the affected IT asset to a user's organization, measured in terms of confidentiality, integrity, and availability. That is, if an IT asset supports a business function for which Availability is most important, the analyst may assign a higher value to Availability relative to Confidentiality and Integrity. Each security requirement has three possible values: Low, Medium, or High. The list of possible values is in table. For the sake of brevity, the same table is used for the three units of measurement. The greater the security requirement, the higher the score ("average" is considered the default value).

Metric value Description
Not Defined (X) Assigning this value to the metric does not affect the score. It signals the equation to skip this metric.
High (H) Loss of [confidentiality/integrity/availability] is likely to have a catastrophic negative effect on the organization or individuals associated with the organization (e.g. employees, customers).
Medium (M) Loss [Confidentiality/Integrity/Availability] is likely to have a serious adverse effect on the organization or individuals associated with the organization (e.g., employees, customers).
Low (L) Loss [Confidentiality/Integrity/Availability] is likely to have only a limited adverse effect on the organization or individuals associated with the organization (e.g. employees, customers).

Modified Base Metrics

These metrics allow the analyst to adapt the Base metrics to the changes that exist within the analyst's environment. That is, if an environment has made general changes to the affected software that affect its exploitability, scope, or impact, then the environment can reflect this through a custom environment score. The list of possible values is presented in table .

Metric value Description
Modified Attack Vector (MAV) Modified Attack Complexity (MAC) Modified Privileges Required (MPR) Modified User Interaction (MUI) Modified Scope (MS) Modified Confidentiality (MC) Modified Integrity (MI) Modified Availability (MA) The same values as the corresponding Base Metric (see Base Metrics above), as well as Undefined (the default).

Vector String

The vector set CVSS v3.0 is a text representation of a set of CVSS statistics. It is often used to capture or transfer CVSS metric information in a concise form. All basic data should be contained in a vector sequence. Temporal and Environmental metrics are optional, and omitted metrics are considered Not Defined (X). Metrics with a value of Not Defined can be explicitly included in a vector string if desired. Programs that read v3.0 vector strings must accept metrics in any order and treat unspecified Temporal and Environmental as Not Defined. A vector string cannot contain the same metric more than once.
